# Release signing: key ceremony and handling

> Status: **no release keys exist yet.** Test builds use AOSP test-keys and
> must never be re-locked onto a device. The ceremony below is executed
> once, before the first Phase 2 signed release; this doc exists first so
> the procedure is reviewed before it is load-bearing.

## Ceremony procedure

1. **Machine:** a laptop with no persistent storage of the secrets —
   booted from a live Linux USB, network hardware disabled (rfkill +
   no cable) for the entire ceremony. Nothing about the ceremony machine
   is reused.
2. **Witnesses:** minimum two people present end-to-end while the project
   has ≥2 maintainers; the owner alone until then, with the session
   screen-recorded to an offline medium.
3. **Generate:** AOSP `development/tools/make_key` for each APK key
   (RSA-2048 as upstream tooling expects, subject
   `/C=US/O=MatzPhone/CN=<keyname>`); `avbtool generate` / openssl
   RSA-4096 for the AVB key. All keys passphrase-protected; passphrases
   generated by diceware at the ceremony, never typed into any networked
   machine afterward.
4. **Store:** two encrypted copies on separate hardware (USB media from
   different manufacturers), stored in separate physical locations, plus
   a paper backup of the AVB key and passphrases. An inventory sheet
   (public key fingerprints only) is committed to this repo after the
   ceremony.
5. **Destroy:** RAM-only working copies end when the machine powers off;
   the live USB is wiped.

## Operational rules

- **CI never holds private keys.** Release signing runs on a dedicated
  offline machine: CI produces `target_files`, a human moves them to the
  signing machine, `sign_target_files_apks` + `ota_from_target_files`
  run there, and only signed artifacts move back.
- Public key fingerprints are published (repo + site) so users can verify
  what their device trusts.
- **Rotation/compromise:** APK keys support rotation via APK Signature
  Scheme v3 lineage; the AVB key cannot be rotated on locked devices
  without a reflash — treat its storage with corresponding paranoia.
  Any suspected compromise: publish an advisory via SECURITY.md channels
  within 72 hours, ship a rotated release, and document the timeline.
- The signing machine's checklist lives with the ceremony records, not
  in CI, so it cannot drift silently.
